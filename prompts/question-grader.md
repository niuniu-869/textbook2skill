# Benchmark 评分 prompt 模板（可选 — 用于 LLM-as-judge 兜底）

`bench.py` 的 `grade()` 函数主要靠规则评分（MCQ 字母提取 + 数字 ±2% + 文本模糊匹配）。

但有些边缘情况规则评分会判错（比如填空题答案是"贡献毛益率"但 LLM 回了"边际贡献率"且 aliases 没穷举到）。这时可选用 LLM-as-judge 兜底。

**默认不启用**——规则评分快、便宜、可解释；LLM-as-judge 慢、贵、引入新的不确定性。但用户怀疑规则评分误判时可以手动跑这个 prompt 复核。

## prompt（手动复核单题）

```
你是一位资深 {领域} 教师。请判断学生的回答是否正确。

题目: {question}
正确答案: {correct_answer}
学生回答: {student_answer}

判断规则:
1. 选择题: 学生回答的字母 == 正确答案的字母 → 正确
2. 填空题数字: 学生数字 在正确数字 ±2% 范围内 → 正确
3. 填空题文本: 学生回答含正确答案关键词或同义表达 → 正确
4. 学生答错了但展示了正确思路 → 仍算错（这是 benchmark，不是平时作业评分）

输出严格 JSON（不要 markdown）:
{
  "correct": true/false,
  "reason": "1 句话说明判断依据"
}
```

## 调用要点（不要在主流程默认启用）

- 默认 `bench.py` 用规则评分，速度快
- 如果用户怀疑某些题被规则误判，单独跑这个 prompt 复核
- LLM-as-judge 30 题串行 ~5 分钟，并发 ~30 秒

## 何时用 LLM-as-judge

| 场景 | 用规则 | 用 LLM-as-judge |
|------|--------|----------------|
| 选择题 | ✓ | 不需要（字母对比稳定） |
| 数字填空 | ✓ | 不需要（±2% 容差稳定） |
| 简单文本填空（"贡献毛益"） | ✓ | 不需要（aliases 够用） |
| 复杂文本填空（含同义词、缩略） | ⚠️ 可能误判 | 推荐用 |
| 计算过程评分 | ❌ | 必须用 |

## 实际产品中的取舍

V0 全部用规则评分。30 题里规则误判约 1-2 题（凭经验估），不影响整体结论。

如果要做更严谨的 benchmark，可以：
1. 规则评分跑一遍拿 baseline
2. 对 WITH 错 / WITHOUT 对的题（disagreement）跑 LLM-as-judge 复核
3. 只对 disagreement 用 LLM-as-judge，省成本
